# Todo 仪表盘设计文档

## 1. 背景

当前项目已经有一个纯前端的「四象限 Todo」页面，但核心体验偏任务分类，不够贴近日常待办记录。此次需求是在现有单页页面中实现一个更通用的 TodoList，满足以下目标：

- 默认以普通 Todo 列表承载日常记录。
- 支持切换到四象限视图，复用同一份任务数据。
- 本次仅实现前端演示，不做 `localStorage` 或后端持久化。

## 2. 目标与范围

### 2.1 目标

- 提供一个可直接使用的 Todo 仪表盘页面。
- 支持新增、完成、删除、筛选任务。
- 支持在「列表视图」与「四象限视图」之间切换。
- 在同一页面展示任务统计，强化任务反馈。

### 2.2 非目标

- 不接入数据库或服务端接口。
- 不实现刷新后保留数据。
- 不实现截止时间、标签、优先级、拖拽排序、回收站等扩展能力。

## 3. 页面结构

页面采用仪表盘结构，保持单页实现：

1. 顶部标题区
   - 展示页面标题、副标题、今日日期。
   - 提供「列表视图 / 四象限视图」切换按钮。

2. 快速录入区
   - 提供任务标题、备注、状态、象限字段。
   - 作为新增任务的主入口，始终保持可见。

3. 统计卡片区
   - 展示总任务数、进行中任务数、已完成任务数、完成率。
   - 所有统计数据基于当前任务数组实时计算。

4. 主内容区
   - 在列表视图下，左侧（或主区域）展示 Todo 列表，支持状态筛选。
   - 右侧展示四象限缩略看板，帮助用户快速感知任务分布。

5. 四象限独立视图
   - 切换后主区域展示完整四象限布局。
   - 每个象限显示任务数量与任务卡片。
   - 与列表视图使用完全相同的数据源。

## 4. 数据模型

任务模型统一为如下结构：

```js
{
  id: string,
  title: string,
  desc: string,
  status: 'todo' | 'done',
  quadrant: 'q1' | 'q2' | 'q3' | 'q4',
  createdAt: string
}
```

字段约束如下：

- `id`：使用浏览器原生唯一标识生成方式。
- `title`：必填，去除首尾空格后不能为空。
- `desc`：选填，作为任务补充说明。
- `status`：仅保留 `todo` 与 `done` 两种状态，避免本次需求膨胀。
- `quadrant`：任务默认归属一个象限，供四象限视图与统计面板使用。
- `createdAt`：用于展示创建时间。

## 5. 交互设计

### 5.1 新增任务

- 用户通过表单输入标题、备注、状态、象限后提交。
- 提交成功后，任务插入列表顶部。
- 表单重置为默认值，便于连续录入。

### 5.2 标记完成

- 列表视图与四象限视图中的任务卡片都支持切换完成状态。
- 完成状态会影响：
  - 卡片视觉样式；
  - 列表筛选结果；
  - 统计卡片中的进行中、已完成与完成率。

### 5.3 删除任务

- 用户可从任一视图删除任务。
- 删除后列表、统计、四象限分布同步更新。

### 5.4 状态筛选

- 列表视图提供 `全部`、`进行中`、`已完成` 三个筛选项。
- 筛选仅影响列表展示，不改变底层数据。

### 5.5 视图切换

- 用户可在页面顶部切换「列表视图 / 四象限视图」。
- 切换视图不会丢失表单输入之外的任务状态。
- 两个视图展示同一份 `tasks` 数据，避免双份数据不一致。

## 6. 视觉设计

视觉风格延续现有轻量效率工具方向，但从「四象限主导」改为「仪表盘主导」：

- 顶部与统计卡片建立首屏层次。
- 列表作为主工作区，强调高频操作。
- 四象限保留颜色区分，但不再独占首页主视觉。
- 在移动端改为单栏堆叠，优先展示快速录入、统计与列表。

## 7. 实现边界

考虑当前仓库只有一个 `index.html` 单页文件，本次实现继续维持单文件 HTML/CSS/JS 结构，不额外引入框架或打包工具。

实现时允许在单页内部按职责拆分函数，包括但不限于：

- 任务数据处理函数；
- 统计计算函数；
- 列表渲染函数；
- 四象限渲染函数；
- 事件绑定函数。

## 8. 测试与验收标准

本次以页面行为验证为主，验收标准如下：

1. 可以新增任务，且新增内容会出现在列表与四象限中。
2. 可以切换 `全部 / 进行中 / 已完成` 筛选，列表结果正确。
3. 可以标记任务完成，统计卡片会同步刷新。
4. 可以删除任务，列表与四象限中的对应任务消失。
5. 可以切换列表视图与四象限视图，两边展示同一份任务数据。
6. 在窄屏下页面不发生关键操作区域溢出，新增、筛选、切换仍可操作。

## 9. 风险与后续扩展

本次明确接受以下限制：

- 页面刷新后数据丢失。
- 任务模型较轻，不支持复杂项目管理能力。

如果后续扩展，可按优先级考虑：

1. 接入 `localStorage` 作为最小持久化方案。
2. 增加截止时间、优先级与标签。
3. 增加搜索、排序与拖拽。
4. 接入后端 API，实现跨设备同步。
